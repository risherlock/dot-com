k-vector range search algorithm

May 5, 2023

k-vector is a time efficient range search algorithm developed by Daniele Mortari for identification of star patterns by star trackers onboard spacecrafts. The name of the algorithm derives from the vector \(\boldsymbol{k}\); one dimensional vector of integers that plays central part in the execution of algorithm.

This blog is intended to be commentary and guide to the paper [1]. I hope this helps you in your journey to understand, or write software, for the algorithm.

Problem statement

In order to better understand the range searching problem, let us start with an example. Suppose there are \(n\) students in a class room and the heights of all students are represented by \(n\) dimensional real vector \(\boldsymbol{y}\). Let us further assume that a fitness instructor wants to find the students whose height lie in range \([y_{a}, y_{b}]\). Now this problem of determining the elements in \(\boldsymbol{y}\) that lie in the given range is called the range search problem. The vector \(\boldsymbol{y}\), on which the search is to be performed, is known as database.

We can solve above problem intuitively by going thorugh each element of the database and comparing if the element lies in the range or not. This works fine but is inefficient if search has to be performed frequently for large \(n\).

We can go further and be more clever. Instead of performing comparison for each elements in the database, we can use better search methods like binary search algorithm. This is good improvement over the former naive approach. However, the binary search algorithm still requires us to perform \(2\)log\({}_{2}n\) comparisions which could be computationally expensive for real-time applications like, for instance, star tracker with its huge database of star information.

The above mentioned problem statement is where k-vector method shines becase it doesnot involve the any comparision based search steps for range finding. It does, however, requires us to create a integer vector \(\boldsymbol{k}\) with same length as of the database (i.e. \(n\)). This memory cost comes with the privilage of its fast execution. Let us now see how the k-vector is constructed and consequent searches are performed.

Algorithm

The variables and notations, which follows from [1], are listed below.

Notations

\(\boldsymbol{y}\) One dimensional, \(n\)-long, vector of real numbers on which the range search is to be performed. Database.
\(\boldsymbol{s}\) Elements in vector \(\boldsymbol{y}\) arranged in ascending order.
\(\boldsymbol{k}\) \(n\)-long special vector of integers whose element represents the number of elements in \(\boldsymbol{s}\) that are less than. The k-vector
\(y_{\text{min}}\quad\) Minimum element in the database. \(y_{\text{min}} = \boldsymbol{s}[1]\).
\(y_{\text{max}}\) Maximum element in the database. \(y_{\text{max}} = \boldsymbol{s}[n]\).
\(z(x)\) Virtual line that forms the basis of construction of \(\boldsymbol{k}\) where m is slope and \(q\) is the \(y\) intercept. \(z(x) = mx + q \).
\(\lceil x\rceil\) Nearest integer greater than \(x\). Example: \(\lceil1.1\rceil\) = 2.
\(\lfloor x\rfloor\) Nearest integer less than \(x\). Example: \(\lfloor1.9\rfloor\) = 1.

Note: The indices of vector starts with 1 and the elements of vectors are indexed using square brackets.

Principle

\(\boldsymbol{s}\) is the sorted database, so if we plot the elements of \(\boldsymbol{s}\) with their corresponding indices as x-axis, we can expect the graph to increase monotonically.

The idea is introduce a line using the end points that would enclose \(y_{min}\) and \(y_{max}\). In order to enclose these points, we extend the line by smallest possible value i.e. the relative machine precision \(\epsilon\). Hence, the virtual line passes through the points \((1,\,y_{min}-\epsilon)\) and \((n,\,y_{max} + \epsilon)\) so that \(y_{min}\) and \(y_{max}\) are enclosed by the end points.

k-vector construction

The algorithm proceeds with construction of k-vector which, in turn, depends on construction of virtual line \(z(x)\) discussed in above section. Since we know the two points that lie on the line (i.e. \((1,\,y_{min}-\epsilon)\) and \((n,\,y_{max} + \epsilon)\)), we can simply use the equation of slope to find its slope \(m\) and y-intercept \(q\). $$m=\frac{y_{\text{max}}-y_{\text{min}}+2\xi}{n-1}$$ and $$q=y_{\text{min}}-m-\xi$$

With the line \(z(x)\) parameterized succssfully, we can use it to construct the k-vector as follows $$\boldsymbol{k}[i]=j$$

where, \(\boldsymbol{s}(j)\leq z(i) \leq\boldsymbol{s}(j+1)\), \(i = 2\text{ to }n-1\), \(\boldsymbol{k}[1]=0\), and \(\boldsymbol{k}[n]=n\).

We can see that the k-vector holds the value of index \(j\) such that the value of the vector \(\boldsymbol{s}\) for that index is less than \(z(i)\). This means that elements of k-vector represents the number of elements in \(\boldsymbol{s}\) less than \(z(i)\). Now the values \(\boldsymbol{k}(1)=0\), and \(\boldsymbol{k}(n)=n\) makes sense because there are no elements in k-vector below \(z(1)\) and all the elements are enclosed by \(z(n)\).

Search

We are ready to search range after the k-vector is constructed. We are looking for the values in \(\boldsymbol{s}\) that lies in the range \([y_{a},\,y_{b}]\). In other words, we are searching for the two indices that encloses the elements in \(\boldsymbol{s}\) that lies in the range. The indices are computed as \begin{equation} j_{b}=\biggr\lfloor\!\frac{y_{a}-q}{m}\biggr\rfloor \end{equation} and \begin{equation} j_{b}=\biggr\lceil\!\frac{y_{b}-q}{m}\biggr\rceil. \end{equation} The range of values in s in range \([y_{a},\,y_{b}]\) is \([\boldsymbol{s}(k_{\text{start}}),\,\boldsymbol{s}(k_{\text{start}})]\) where the indices \(k_{\text{start}}\) and \(k_{\text{end}}\) are computed as \begin{equation} k_{\text{start}} = k[j_{b}] + 1 \end{equation} and \begin{equation} k_{\text{end}} = k[j_{t}]. \end{equation} Hence, there are \(k_{\text{end}}-k_{\text{start}}\) elements in database which lies in range \([y_{a},\,y_{b}]\) and the elements are \(\boldsymbol{s}[k_{\text{start}}]\), \(\boldsymbol{s}[k_{\text{start}+1}]\), \(\boldsymbol{s}[k_{\text{start}+2}]\), ... , \(\boldsymbol{s}[k_{\text{end}}]\).

An extraneous element

So far we saw that once k-vector is constructed, the search phase is actually some couple of equations and we get our solution. This minimalism is the beauty of k-vector algorithm. However we are not done yet for there is one subtlety that we need to handle.

The problem is that it is not guranteed that the elements represented by \(k_{\text{start}}\) and \(k_{\text{end}}\) lie in the range. There is a 50% probability the each of them doesnot lie in the range. In the cases whereprecision is crucial, so we need to comparision checks to make sure that \(k_{\text{start}}\) and \(k_{\text{end}}\) elements, in fact, lie in the range. The pseudocode for the checks are presented below.

while \(\left(\boldsymbol{s}[k_{\text{start}}]\lt y_{a}\right)\)
{
    \(k_{\text{start}}\)++;
}
 while \(\left(\boldsymbol{s}[k_{\text{end}}]\gt y_{b}\right)\)
{
    \(k_{\text{end}}\)--;
}

This extra step to otherwise computationally minimal search algorithm raises a question. How many times will the while loop go on? \(E_{0}=n/(n-1)\) is the number of checks what we have to perform for each indices. This, however, does not add too much burden to the algorithm because, for large databases where \(n\rightarrow\infty\), \(E_{0}\rightarrow 1\).
